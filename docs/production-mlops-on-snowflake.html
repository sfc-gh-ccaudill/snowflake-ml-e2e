Section 01

Architectural Design

Production ML systems fail not because of the model — they fail because of fragmented infrastructure. Data lives in one system, training happens in another, inference runs somewhere else, and monitoring is bolted on as an afterthought. Snowflake's approach is to unify all of these layers into a single governed platform, dramatically reducing operational overhead while keeping strong security boundaries.

Before choosing an architecture, teams must make a foundational decision about account structure. The three patterns below represent a spectrum from simplest to most operationally isolated — and they are not mutually exclusive. Many large organizations start with Pattern 1, graduate to Pattern 2 as teams grow, and eventually adopt Pattern 3 once ML becomes a first-class operational discipline with dedicated platform infrastructure.

Architecture 1: Single-Account (Unified Dev/Prod)

In the single-account pattern, all ML assets — training data, feature store, model registry, compute pools, and inference endpoints — coexist in one Snowflake account. Environment isolation is achieved through dedicated databases (e.g., ML_DEV and ML_PROD), separate schemas within those databases, role-based access control, and naming conventions. There are no cross-account networking concerns, no data replication delays, and billing is consolidated. This is the right starting point for most teams.

The key to making this work is a strict RBAC hierarchy where data scientists have free rein in development databases but cannot write to production. A dedicated ml_admin role bridges the two environments and is the only principal allowed to copy model versions from dev to prod schemas.

Figure 1 — Single-Account Architecture: Dev & Prod Isolated by Database and Role

✅ Choose single-account when…

  • Team size is under ~50 ML practitioners
  • No hard regulatory requirement for physical isolation
  • Simpler billing consolidation is preferred
  • Fast iteration speed is a priority
  • Single cloud provider and region

⚙️ Isolation mechanism summary

  • Separate databases: ML_DEV and ML_PROD
  • Role hierarchy controls who can write where
  • ml_admin is the only role spanning both databases
  • Compute pools can be scoped per environment
  • Future grants auto-extend access to new models

Architecture 2: Multi-Account (Physically Isolated Dev/Prod)

For enterprises with strict compliance requirements — SOX, HIPAA, FedRAMP, or internal governance policies mandating network-level separation — placing development and production in separate Snowflake accounts provides a hard security boundary. A rogue process or misconfigured permission in dev simply cannot affect the production environment. This architecture is also common in large organizations where separate teams own each environment.

The trade-off is operational complexity: data and model artifacts must move between accounts deliberately. Snowflake provides two mechanisms for this. Account-level replication groups can synchronize entire databases (including model registry schemas) from dev to prod on a schedule. Alternatively, a promotion pipeline authenticates to both accounts and programmatically exports model artifacts from the dev registry, then re-logs them in the prod registry — giving fine-grained control over exactly which versions are promoted and when.

✅ Choose multi-account when…

  • Dedicated teams own dev vs. prod environments separately
  • You require separate cloud regions or cloud providers per environment
  • Separate billing and chargeback structures are required
  • A security breach in dev must be unable to affect prod

Architecture 3: Dedicated ML Operations Account

The most operationally mature pattern treats machine learning infrastructure as a first-class platform that is physically separated from both the data platform and the downstream consumer applications. In this topology, one or more Data Accounts own production data — the authoritative source of truth for operational and analytical data. A dedicated ML Ops Account sits between the data and consumers: it reads from data accounts, runs all training and inference workloads, and exposes models back to consumers. Consumer Accounts (internal analytics and application teams) and external services integrate with the ML Ops account for inference without ever having direct access to raw training data.

This architecture is compelling for organizations with a mature data platform team and a growing ML platform team that needs independent scaling, cost attribution, and governance. The ML Ops account can adopt aggressive GPU capacity without affecting data platform compute costs. Security teams can audit ML operations — who ran training, what data was used, which models were deployed — completely independently from other account activity.

How Data Gets Into the ML Ops Account

The ML Ops account does not store raw production data. Instead, Data Accounts create Secure Data Shares — zero-copy, real-time read-only access grants — which the ML Ops account mounts as databases. From the ML Ops account's perspective, shared tables behave exactly like local tables: they can be queried, joined, and used as the basis for feature engineering. No data is copied or replicated; every read goes directly to the source. This preserves a single copy of truth, simplifies compliance (the data account retains full ownership and audit history), and eliminates the infrastructure overhead of a separate ETL pipeline just to move data into the ML account.

How Models Get Back to Consumers

Model artifacts are distributed to consumers through two complementary channels. For SQL-based inference within other Snowflake accounts — where teams want to call model!PREDICT(...) inline in queries — the ML Ops account uses Account Replication to push a replica of the model registry schema to each consumer account on a defined schedule. Consumer accounts then query against a local copy of the model objects, keeping inference latency low and compute costs in the consumer's own account. For real-time REST inference — from either other Snowflake accounts or non-Snowflake services like web backends, mobile platforms, or partner systems — requests go directly to the HTTPS endpoint exposed by the ML Ops account's SPCS inference service.

Figure 3 — Dedicated ML Operations Account: Data Sharing, Training, and Multi-Consumer Inference

Data Accounts grant Secure Data Shares to the ML Ops account — no data moves, reads go directly to the source. The ML Ops account replicates its model registry schema to Consumer Accounts so teams can call models in SQL. Any HTTP client — Snowflake accounts or external services — can call the SPCS inference endpoint over HTTPS.

ℹ️
Secure Data Sharing vs. Account Replication: The Right Tool for Each Direction

Use Secure Data Sharing when the ML Ops account needs to read from Data Accounts — it is zero-copy, real-time, and read-only. The data never leaves the source account. Use Account Replication when pushing model artifacts from ML Ops to Consumer Accounts — because inference needs a local copy of the model objects to resolve SQL calls like model!PREDICT() within the consumer account's own compute context. The two mechanisms serve opposite data flow directions and are complementary, not interchangeable.

Managing Multi-Account ML Architectures with Snowflake Organizations

As soon as you have two or more Snowflake accounts — whether for dev/prod isolation or a full data + ML ops + consumer topology — the operational overhead of managing them independently grows significantly. Snowflake Organizations are the answer to this: a top-level administrative layer that spans all of your Snowflake accounts under a single governance umbrella.

An Organization is automatically associated with your Snowflake contract. Every account you create — across cloud providers, regions, and environments — belongs to the same Organization. The ORGADMIN system role is the primary management interface. Through it, administrators can create new accounts, rename them, view cross-account credit consumption, and set organization-wide policies that apply uniformly across all accounts.

Figure 4 — Snowflake Organization: Centralized Governance Across All ML Architecture Accounts

With an Organization in place, a number of cross-account capabilities become available that are impractical to manage per-account:

🔐

Org-Level Authentication Policies

Define a single Authentication Policy that requires MFA for all human users across every account in the organization. Service accounts used for CI/CD and model promotion can be explicitly exempted while all interactive users remain enforced.

📊

Cross-Account Cost Visibility

The SNOWFLAKE.ORGANIZATION_USAGE schema provides a unified view of credit consumption, storage, and compute costs across all accounts. ML teams can clearly see GPU compute costs in the ML Ops account separately from warehouse costs in consumer accounts — enabling per-team chargeback models.

🗂️

Replication Group Management

Replication Groups are defined at the organization level and specify which databases (including model registry schemas) replicate to which target accounts. This is the operational mechanism that keeps model versions synchronized from the ML Ops account to consumer accounts.

🛡️

Trust Center Across the Org

The Trust Center provides a security posture dashboard across all accounts in the organization. Security teams can identify unprotected objects, excessive privileges, and missing MFA enrollment from a single view rather than auditing each account independently.

🏗️

Account Lifecycle Management

The ORGADMIN role can create new Snowflake accounts programmatically — useful for spinning up ephemeral experiment accounts, onboarding new regional data platforms, or creating an ML staging account without manual provisioning workflows.

🌐

Cross-Region & Multi-Cloud

An Organization can span accounts across AWS, Azure, and GCP, as well as multiple geographic regions. A Data Account in AWS us-east-1 can share data with an ML Ops Account in AWS us-west-2, or across clouds entirely — enabling compliance with data residency requirements alongside centralized ML infrastructure.

🏢
Architecture Decision Summary: Choosing Your Pattern

Single account — fastest to start, easiest to operate, appropriate for most teams. Multi-account dev/prod — when compliance mandates physical isolation between development and production. Dedicated ML Ops account — when ML infrastructure is a platform in its own right with dedicated cost attribution, independent scaling, and multiple consuming teams. Snowflake Organizations apply to all multi-account patterns and should be enabled from the start: the cost is zero and the operational leverage grows with every account you add.

Section 02

Compute Pools & Remote ML Jobs

All container-based ML workloads in Snowflake run on compute pools — collections of one or more virtual machine nodes managed entirely by Snowflake. Think of a compute pool as the container-native equivalent of a virtual warehouse: you define the machine type and node count, and Snowflake handles provisioning, autoscaling, patching, and teardown. Reference: SPCS: Working with Compute Pools.

Compute pools are account-level objects. Every Snowflake account is automatically provisioned with two system compute pools — one CPU (SYSTEM_COMPUTE_POOL_CPU using CPU_X64_S nodes) and one GPU (SYSTEM_COMPUTE_POOL_GPU using GPU_NV_S on AWS or GPU_NV_SM on Azure). These pools start suspended and incur no cost until a workload uses them, auto-suspending again after 10 minutes of idle time. They are the fastest path to getting started without any administrator setup.

CPU vs. GPU: Choosing the Right Instance Family

Snowflake exposes a rich set of instance families across all three major cloud providers. CPU families range from the lightweight CPU_X64_XS (ideal for development, CI jobs, and light inference) up to HIGHMEM_X64_L with 984 GB of RAM for in-memory workloads. GPU families are available in AWS, Azure, and GCP, covering NVIDIA T4, A10, A10G, A100, and L4 hardware.

Autoscaling and Lifecycle

Compute pools autoscale between their minimum and maximum node counts. When running services demand more capacity than the current nodes can provide, Snowflake provisions additional nodes up to the MAX_NODES limit. Conversely, idle nodes are reclaimed down to MIN_NODES. For production serving pools, setting MIN_NODES = 2 ensures at least two instances survive routine maintenance windows. The PLACEMENT_GROUP = DISTRIBUTED setting instructs Snowflake to spread nodes across availability zones, providing fault tolerance if one zone experiences an outage.

ℹ️
Compute Pool Maintenance Windows

Snowflake replaces compute pool nodes regularly (typically every few weeks) during weekend maintenance windows. Services running on multi-instance pools are recreated incrementally — no more than 50% of instances are replaced at once. Single-instance services will experience brief disruption. For production HA services, set MIN_INSTANCES ≥ 2 and configure a readiness probe so traffic is only routed to fully initialized instances.

Long-Running Services vs. Job Services

Snowpark Container Services supports two operational modes. Services are long-running processes — they keep running until you explicitly stop them and are ideal for online inference endpoints, Streamlit applications, and background processing workers. Job services, triggered with EXECUTE JOB SERVICE, run to completion and terminate — just like a Spark job or a cron script. Training runs, batch scoring jobs, and data preprocessing pipelines all use the job service pattern. Job services integrate naturally with Snowflake Tasks for scheduling and with DAG-based orchestration (see Section 4).

💡
2× Faster Than Managed Spark

Snowflake's benchmarks show distributed XGBoost training in Snowflake Notebooks on Container Runtime is 2× faster on a 5 GB tabular dataset compared to a managed Spark solution, using default settings out of the box. Source: Snowflake Blog, March 2025.

Distributed Training APIs

For large-scale model training, Snowflake provides distributed training APIs that extend the familiar interfaces of XGBoost, LightGBM, and PyTorch without requiring manual orchestration. The XGBDistributed and LGBMDistributed wrappers automatically spread the training workload across all CPUs or GPUs in the compute pool. For PyTorch, Snowflake integrates with PyTorch's DistributedDataParallel under the hood, allowing multi-node multi-GPU training from a standard notebook. Similarly, the DataConnector API parallelizes data loading directly from Snowflake tables, eliminating the bottleneck of serial pandas reads for large datasets.

Ray on SPCS: Distributed ML at Scale

Ray is an open-source distributed computing framework that complements Snowflake's native training APIs for workloads requiring fine-grained control over parallelism — large-scale hyperparameter search with Ray Tune, multi-node distributed training with Ray Train, and scalable data ingestion from Snowflake tables with Ray Data. Ray clusters run natively on SPCS: the Ray head node runs as a long-running Service, and worker nodes run as additional instances in the same compute pool, discovering each other automatically via SPCS service DNS without any external cluster manager.

The integration with the rest of the Snowflake ML stack is clean. A Task DAG step submits a job to the Ray cluster via the Ray Jobs API, blocks until completion, and surfaces the exit code back to the Task — so Ray training runs participate in Task retry logic, execution history, and error propagation identically to any other pipeline step. The training script itself calls Registry.log_model() from snowflake-ml-python, so the best trial's artifact lands in the Model Registry in the same way as a single-GPU training run.

-- Step 1: launch the Ray cluster as a long-running Service
-- SPCS assigns a stable DNS name (ray-cluster) that workers use to join
CREATE SERVICE ray_cluster
  IN COMPUTE POOL GPU_POOL
  FROM SPECIFICATION $$
    spec:
      containers:
        - name: head
          image: /ml_ops/registry/ray-training:v1.2
          command: ["ray", "start", "--head",
                    "--port=6379", "--dashboard-host=0.0.0.0", "--block"]
          resources:
            requests: {nvidia.com/gpu: "1"}
        - name: worker
          image: /ml_ops/registry/ray-training:v1.2
          command: ["sh", "- c",
                    "ray start --address=ray-cluster:6379 --block"]
          resources:
            requests: {nvidia.com/gpu: "1"}
      numInstances: 5   # 1 head + 4 GPU workers
  $$;

-- Step 2: submit the HPO job from a Task DAG step
-- Job blocks until Ray job completes; non-zero exit propagates to Task
EXECUTE JOB SERVICE
  IN COMPUTE POOL CPU_POOL
  FROM SPECIFICATION $$
    spec:
      containers:
        - name: submitter
          image: /ml_ops/registry/ray-training:v1.2
          command: ["ray", "job", "submit",
                    "--address", "http://ray-cluster:8265",
                    "--", "python", "tune_hpo.py",
                    "--trials=20", "--gpus-per-trial=1"]
  $$;

⚡
Ray Tune: Parallel HPO at the Cost of a Single Training Run

A GPU_NV_M node (4× A10G GPUs) can run 4 concurrent Ray Tune trials. Across 5 worker nodes that’s 20 simultaneous trials in the same wall-clock time a single sequential trial would take. Each trial logs its metrics and artifact to the Model Registry as a separate model version via Registry.log_model(), and the best-performing version is selected by the downstream gate step using the same evaluation logic as any other training run.

Ray is particularly well-suited for three scenarios in a Snowflake ML pipeline: (1) hyperparameter optimization where the search space is too large for sequential tuning, (2) LLM fine-tuning that requires coordinating gradient synchronization across multiple GPU nodes, and (3) reinforcement learning workloads where distributed environment simulation is the bottleneck. For standard tabular training and feature-scale distributed XGBoost, Snowflake’s native XGBDistributed API is simpler and requires no cluster management overhead.

Section 03

CI/CD for Training Containers

Production ML requires treating training code with the same rigor as application software. Every training container must be version-controlled, reproducibly buildable, tested on every commit, and deployable without manual steps. Ad-hoc notebook executions and manually pushed Docker images are the leading cause of "it worked last week" failures in ML systems.

Snowflake integrates natively with popular CI/CD systems through the snowflake-cli-action GitHub Action. This action handles Snowflake CLI installation, authentication configuration, and allows running any snow command as part of your GitHub workflow — including pushing images to the SPCS container registry, executing job services, and deploying updated service specifications.

Pipeline Overview

Figure 5 — CI/CD Pipeline for ML Training Containers

Authentication: OIDC over Static Keys

The recommended authentication pattern for CI/CD is Workload Identity Federation (OIDC), available in Snowflake CLI version 3.11+. Instead of storing a Snowflake private key as a GitHub secret, OIDC uses GitHub's ephemeral identity tokens — issued fresh for each workflow run — to authenticate with Snowflake. There are no long-lived credentials to rotate or leak. The setup requires creating a SERVICE user in Snowflake with a workload identity bound to your GitHub repository and environment name.

For teams not yet on Snowflake CLI 3.11, private key authentication is the fallback. Store the key as an encrypted GitHub secret and reference it as an environment variable in the workflow. Never use password authentication in CI; it cannot be restricted to specific workflows or IP ranges.

# .github/workflows/deploy.yml — core pattern (abbreviated)
name: Build and Deploy Training Container
on:
  push:
    branches: [main]
    paths: ["training/**"]

permissions:
  id-token: write    # required for OIDC authentication
  contents: read

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Set up Snowflake CLI with OIDC
        uses: snowflakedb/snowflake-cli-action@v2.0
        with:
          use-oidc: true
          cli-version: "3.11"
        env:
          SNOWFLAKE_ACCOUNT: ${{ secrets.SNOWFLAKE_ACCOUNT }}

      - name: Build, push, and smoke test
        run: |
          snow spcs image-registry login
          docker build --platform linux/amd64 -t $REGISTRY/trainer:${{ github.sha }} ./training
          docker push $REGISTRY/trainer:${{ github.sha }}
          snow spcs job execute --spec jobs/smoke_test.yaml --compute-pool ml_cpu_pool
YAML

Container Best Practices for SPCS

All SPCS containers must target the linux/amd64 architecture regardless of the developer's local machine. Containers should run as a non-root user for security. Pin every dependency to an exact version in requirements.txt — a container that produces different results because a transitive dependency was updated is worse than a container that fails to build. Embed the Git SHA as a Docker build argument and label so any running container can be traced back to the exact source commit that produced it.

For GPU containers, use the official NVIDIA CUDA base images that match the driver version available on your chosen instance family. Snowflake's pre-built notebook images already handle this, but custom training containers need to handle it explicitly.

Section 04

Pipeline Development, Task Graphs & Migration to Production

A production training pipeline is significantly more than a single model training call. It includes data validation, feature extraction, training, evaluation against a holdout set, performance gating, and registry logging — and all of these steps need to be individually testable, resumable on failure, and auditable. Snowflake provides two complementary environments for building this: an interactive development environment via Notebooks on Container Runtime, and a production-grade orchestration layer via Task DAGs.

Development: Snowflake Notebooks on Container Runtime

Snowflake Notebooks on Container Runtime are generally available on AWS and in public preview on Azure. They run on compute pools — including the default system pools that require no administrator setup — and come pre-installed with the most popular ML frameworks: PyTorch, XGBoost, LightGBM, scikit-learn, Hugging Face Transformers, and more. PyPI integration allows installing any additional package securely without leaving the Snowflake perimeter.

The development experience is purpose-built for ML: data scientists select a compute pool (CPU or GPU) directly in the notebook UI, and the system handles all infrastructure provisioning. Optimized data ingestion APIs load Snowflake tables in parallel across multiple CPUs or GPUs as pandas or PyTorch DataFrames, eliminating the serial bottleneck of standard ODBC reads. Distributed training APIs wrap XGBoost and PyTorch with automatic multi-node distribution — no cluster configuration required.

Notebooks are the right environment for exploration, iteration, and experiment tracking. Once a training approach is validated, the pipeline logic is extracted into a standalone Python script, containerized, and handed to the orchestration layer.

Production Orchestration: Task DAGs

Snowflake Tasks are serverless scheduled units of work — they can execute SQL, call stored procedures, or trigger SPCS job services. When multiple Tasks are wired together with explicit dependencies, they form a Directed Acyclic Graph (DAG) — a structured pipeline where each step runs only after its predecessor completes successfully. This is the production-grade pattern for orchestrating ML training pipelines entirely within Snowflake, without an external orchestrator like Airflow or Prefect.

A typical ML training DAG includes steps for data freshness validation, feature computation, model training, offline evaluation, and conditional promotion to the model registry. Each Task in the DAG can be configured with its own warehouse size or compute pool, retry policies, and error handling. If evaluation fails (accuracy below threshold, PSI too high), the DAG can halt without promoting a bad model, and the failure is surfaced in Snowflake's task history with full logs.

Figure 6 — ML Training Pipeline as a Snowflake Task DAG

Task DAGs provide several advantages over external schedulers for ML pipelines. All execution history, logs, and task run metadata are stored natively in Snowflake and queryable via TASK_HISTORY(). Tasks participate in Snowflake's access control model — the same RBAC that governs data access governs who can create and resume tasks. There is no external infrastructure to maintain, and scheduling integrates directly with Snowflake Alerts, enabling event-driven triggers (e.g., "retrain when data volume exceeds 10M new rows" or "retrain when drift is detected").

⚡
Substituting Ray for the TRAIN MODEL Step

For workloads requiring parallel hyperparameter search or multi-node distributed training, the EXECUTE JOB SERVICE step in the DAG can submit a job to a Ray cluster running on SPCS (see Section 2). The submitter container blocks until the Ray job completes and propagates the exit code back to the Task — preserving retry and error-handling behavior while enabling 20+ parallel HPO trials as a transparent swap. The training script calls Registry.log_model() to log the best artifact, making the Ray run invisible to the downstream gate and promotion steps.

Dev-to-Prod Migration Strategy

The migration from a notebook experiment to a production pipeline follows a clear progression. In development, the data scientist works interactively in a notebook, iterating on feature engineering and model architecture. Once satisfied, the training logic is extracted into a Python script with environment variables replacing hardcoded database/schema references. The script is containerized, pushed through CI, and run as a job service on a dev compute pool. Once the job service produces consistent results matching the notebook, the production task DAG is created pointing at the same container image (a release-tagged version), and the pipeline is activated in the production environment.

Section 05

Model Versions & Promotion to Production

The Snowflake Model Registry treats machine learning models as first-class schema-level objects — governed by the same RBAC, tagging, lineage, and information schema as tables and views. Each model can have unlimited versions. One version is designated as the default: when production code calls a model by name without specifying a version, it always executes against the default. Changing the production model requires only updating the default version pointer — no application redeployment needed.

The registry stores every logged model's serialized weights, environment specification (conda or pip dependencies), function signatures, and associated metrics. It is generally available as of snowflake-ml-python version 1.5.0. Reference: Managing Models in the Registry.

Single-Account Promotion Flow

In the single-account pattern, a model is trained and logged into the dev registry. An ml_admin role — which has READ access to the dev registry and OWNERSHIP of the prod registry — copies a specific version from dev into prod using the CREATE MODEL … FROM MODEL or ALTER MODEL … ADD VERSION FROM MODEL SQL commands. No model artifacts leave the Snowflake account; the copy is efficient and governed. Once in prod, setting the default version makes the promotion live.

Figure 7 — Single-Account Model Promotion Workflow

Multi-Account Promotion Flow

Cross-account model promotion requires a deliberate mechanism because Snowflake's CREATE MODEL … FROM MODEL syntax operates within a single account. Two approaches are commonly used:

Option A — Export and Re-log: An ml_admin service user — which has credentials for both the dev and prod accounts — exports the model artifacts from the dev registry using mv.export(), which writes the serialized model weights and metadata to a local or stage path. The same user then opens a session in the prod account and re-logs the model using reg.log_model() with an identical configuration. This gives full control over every field and is the most explicit approach.

Option B — Database Replication: Snowflake's Account Replication Groups can synchronize an entire database — including the model registry schema — from the source account to a replica in the destination account. The receiving team then has a read-only mirror of all source registry objects and can selectively expose specific versions. This is lower operational overhead but less granular.

Figure 8 — Multi-Account Model Promotion: Export & Re-log Pattern
⚠️
Never replace a prod version in place

Always add a new version rather than updating an existing one. Model version artifacts are immutable once logged — this is by design. If you need to make a small fix, log a new version with an incremented name. This preserves full reproducibility: you can always re-run inference with any historical version against any historical dataset.

Section 06

Inference Endpoints & Traffic Migration

Snowflake supports three distinct inference patterns. The right choice depends on latency requirements, throughput characteristics, and whether the consuming application is inside or outside Snowflake.

🏭

Warehouse Batch Inference

Call a model directly from SQL using virtual warehouse compute. No infrastructure management. Ideal for nightly scoring, BI enrichment, and any workload that can tolerate latency in the seconds-to-minutes range. The DEFAULT version pointer means no SQL changes are needed when models are promoted.

⚡

SPCS Online REST Endpoint

Deploy any model version as a managed HTTP service with a stable public HTTPS endpoint. Autoscales across instances, supports GPU compute, and is fully integrated with Snowflake authentication. Sub-second latency for user-facing applications. GA since snowflake-ml-python 1.25.0. See: REST Inference Docs.

📦

SPCS Batch Job Service

For large-scale batch scoring that exceeds warehouse capacity — especially GPU-accelerated inference on millions of rows — run a job service on a GPU compute pool. The job reads from a Snowflake table, scores in parallel, writes results back, and terminates. No long-running service cost.

Online Serving Architecture

When you call mv.create_service(), Snowflake automatically builds a container image from the model artifacts, provisions instances on the specified compute pool, and exposes a public HTTPS endpoint. The endpoint URL follows the format unique-id.snowflakecomputing.app. Authentication uses Snowflake Programmatic Access Tokens (PAT) passed in the Authorization header — there are no API keys to manage separately from Snowflake's existing identity model.

The service configuration controls the trade-off between cost and availability. Setting min_instances=2 with PLACEMENT_GROUP=DISTRIBUTED on the underlying compute pool ensures the service survives a single-zone outage. For GPU models, the recommended pattern is to use the smallest GPU node family that fits the model into VRAM, then scale horizontally by increasing max_instances rather than vertically with a larger node.

Traffic Migration: The Canary Pattern

Never flip 100% of production traffic to a new model version immediately. Instead, deploy the new version as a separate SPCS service alongside the stable version. Route a small fraction of traffic (5–10%) to the canary service via your application's feature flag or load-balancing logic. Monitor both services with ML Observability for 24–48 hours. If the canary's drift, accuracy, and error rate metrics are comparable to the stable version, gradually shift more traffic until the canary becomes the new stable service and the old service is decommissioned.

Figure 9 — Canary Deployment: Gradual Traffic Migration Between Model Versions

Once a model version is fully promoted, update the default version in the registry so that SQL batch inference and any code calling the model by name automatically uses the new version. Decommission the old SPCS service using DROP SERVICE only after confirming the new service is healthy and the rollback window has passed.

Section 07

ML Observability

ML Observability is Snowflake's built-in model monitoring system, generally available in all commercial regions. It operates on inference data stored in Snowflake tables — regardless of whether the model was trained inside Snowflake or imported from an external system — and does not require any additional infrastructure. Every model version in the registry can have an associated monitor that refreshes on a configurable schedule and populates a monitoring dashboard directly in Snowsight.

How It Works

When you deploy a model for inference, you log the inputs, predictions, and (eventually) the ground-truth outcomes to a Snowflake table. A model monitor is then created against that table, specifying which columns represent features, predictions, and actuals. Snowflake refreshes the monitor on the configured schedule, computing drift metrics against a baseline dataset and updating performance metrics as ground-truth labels arrive. The results are surfaced in a Snowsight dashboard and are queryable via SQL metric functions for custom reporting.

Figure 10 — ML Observability Workflow

What Gets Monitored

ML Observability measures three categories of metrics for each model monitor. Performance metrics track prediction quality over time — accuracy, AUC, F1, and RMSE — computed as ground-truth labels arrive in the log table. Drift metrics measure how the input feature distributions and prediction distributions have shifted relative to the training baseline, using statistical tests like Population Stability Index (PSI) and Kolmogorov-Smirnov (KS). Statistical metrics track data quality signals like prediction volume, null rates, and out-of-range values.

All three metric categories support segmentation — you can compute any metric broken down by a string categorical column in your data (e.g., by region, customer tier, or product category). This is critical for detecting demographic-specific model degradation that would be masked in aggregate metrics. Each monitor supports up to 5 segment columns with up to 25 unique values each.

📊
PSI Threshold Guidance

A Population Stability Index (PSI) below 0.1 indicates no meaningful distribution shift. PSI between 0.1–0.2 signals moderate shift worth investigating. PSI above 0.2 is widely accepted in the industry as indicating significant drift that typically warrants retraining. These thresholds should be tuned per use case — a fraud model may need tighter thresholds than a recommendation model.

Automated Retraining Loop

Connecting monitoring to automated retraining is straightforward using Snowflake's native alerting. A Snowflake Alert evaluates a SQL condition on a schedule — for example, checking whether any feature's PSI has exceeded 0.2 in the last 24 hours. When the condition is true, the alert executes an action: typically a EXECUTE TASK call that triggers the retraining DAG described in Section 4. The retrained model is then evaluated against the same quality gate, and if it passes, it is promoted to production and the monitors are updated to point to the new version.

Figure 11 — Closed-Loop Automated Retraining Pipeline

Limitations to Be Aware Of

ML Observability currently supports regression and binary classification models. Multi-class classification and ranking models are not yet supported as first-class task types for monitoring. A maximum of 250 monitors can be created per account, and each monitor can track up to 500 features. Segment columns must be string categorical types and should have fewer than 25 unique values for optimal performance. Drift calculation requires a baseline dataset; if no baseline was provided at monitor creation time, the monitor must be recreated to add one.

Section 08

ML Lineage

ML Lineage is generally available in all regions and provides end-to-end traceability of the entire ML data flow — from raw source tables through feature engineering, to the dataset used for training, to the model version produced, and through to inference outputs. Unlike traditional data lineage tools that only track table-to-table transformations, ML Lineage understands the semantics of ML-specific objects: feature views, datasets, and model versions are all first-class nodes in the lineage graph.

The Lineage Graph

Every object created through Snowflake ML automatically participates in lineage tracking without any additional configuration. When you create a feature view over a source table, Snowflake records that dependency edge. When you create a training dataset from feature views, those edges are captured. When you log a model using reg.log_model() with a reference to the training data, the model version is linked to the dataset that produced it. The resulting graph lets any team member — or an auditor — trace any model prediction back through the pipeline to its original data sources.

Figure 12 — ML Lineage Graph: From Raw Data to Production Inference

Key Use Cases for ML Lineage

🔍

Root Cause Debugging

When a model's predictions degrade unexpectedly, lineage lets you trace the problem upstream. Did a feature table's schema change? Did an upstream ETL job start producing nulls? Lineage shows the exact path from symptom to cause.

🏛️

Regulatory Compliance & Audit

Regulators (GDPR, CCPA, SR 11-7, model risk management) often require proof of what data trained a model and when. Snowflake lineage provides a queryable, immutable audit trail that answers these questions without custom metadata tracking.

💥

Impact Analysis

Before changing a source table or feature view, lineage shows every downstream model that would be affected. Teams can coordinate changes rather than discovering breakages after the fact in production.

🔁

Reproducibility

Because the lineage graph captures the exact feature view version and dataset snapshot that trained each model version, any historical model can be precisely reproduced — critical for A/B testing, challenger models, and post-incident analysis.

Lineage and the Snowflake Feature Store

ML Lineage integrates deeply with the Snowflake Feature Store. Feature views registered in the feature store carry their own lineage edges to the underlying source tables. When a dataset is materialized from feature views using the Dataset API, those edges are automatically propagated to the training dataset node. This creates a fully automated, end-to-end lineage graph from raw ingestion tables to live model predictions — without any manual annotation.

Section 09

Governance, Security & Best Practices

Snowflake's ML capabilities are built on the same governance foundation that governs all other Snowflake objects. Models participate fully in role-based access control, information schema queries, object tagging, access history, and the Snowflake Horizon catalog. There is no separate security model to learn or administer for ML workloads.

Model Access Control

Models have three distinct privilege levels. OWNERSHIP grants full control — the owning role can manage versions, update metadata, and access model artifacts (weights, serialized files). USAGE grants the ability to run warehouse-based inference without seeing any internals — appropriate for production application roles. READ grants access to model artifacts and metadata in addition to SPCS inference — appropriate for teams that need to inspect or export model weights.

The recommended pattern is to grant FUTURE USAGE ON MODELS IN SCHEMA ML_PROD.REGISTRY TO ROLE ml_inference so that every newly promoted model is automatically accessible to the inference role without any manual grant. This prevents the common operational failure mode where a newly promoted model is inaccessible to production services because someone forgot to run a grant command.
